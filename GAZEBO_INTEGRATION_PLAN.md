# Haptic Suit Gazebo Integration Plan

This document outlines the plan to expand the Haptic Suit project by integrating real-time forces from a Gazebo simulation.

## High-Level Goal

The objective is to pipe forces generated by a simulated drone in Gazebo through ROS/Mavros and use them to drive the haptic suit, replacing the current predefined force patterns.

## The Plan

The integration is broken down into three main phases:

### Phase 1: Gazebo Simulation and Force Extraction

The first phase focuses on setting up the simulation environment and extracting the necessary force data.

1.  **Set up Gazebo SITL Environment**:
    *   Create a Gazebo world with a suitable drone model (e.g., Iris quadcopter).
    *   Configure a Software-In-The-Loop (SITL) instance of a flight controller (e.g., ArduPilot or PX4) to control the simulated drone.
2.  **Connect Mavros to SITL**:
    *   Bridge the communication between the SITL flight controller and ROS using Mavros. This will allow ROS nodes to subscribe to topics published by the flight controller.
3.  **Extract Actuator Forces**:
    *   Identify and subscribe to the relevant Mavros topic that publishes the drone's motor actuator signals. The `/mavros/actuator_control` topic is a likely candidate, publishing `mavros_msgs/ActuatorControl` messages.

### Phase 2: Create a ROS Node for Force Translation

A new ROS node will be created to process the raw simulation data and translate it into a format the haptic suit can understand.

1.  **Create `gazebo_force_translator.py` Node**:
    *   A new ROS node script will be created at `drone_ws/src/flight_controller/src/gazebo_force_translator.py`.
2.  **Subscribe to Actuator Data**:
    *   This node will subscribe to the actuator data topic from Mavros (e.g., `/mavros/actuator_control`).
3.  **Implement Translation Logic**:
    *   The core of this node will be the logic to convert the raw motor signals from the simulation into the `[angle1, angle2, throttle1, throttle2]` format required by the haptic suit. This will involve mapping, scaling, and potentially filtering the data.
4.  **Publish Haptic Commands**:
    *   The translated commands will be published as a `std_msgs/Float64MultiArray` message on a new ROS topic, for example, `/haptic_suit/gazebo_control_signal`.

### Phase 3: Adapt the Haptic Suit Controller

The final phase involves creating a new haptic suit controller that listens for commands from the new translator node.

1.  **Create `suit_controller_gazebo.py`**:
    *   A new controller script, `drone_ws/src/flight_controller/src/suit_controller_gazebo.py`, will be created, likely based on `suit_controller_stable.py`.
2.  **Subscribe to Gazebo Commands**:
    *   This new controller will subscribe to the `/haptic_suit/gazebo_control_signal` topic.
3.  **Send Commands to the Suit**:
    *   The controller will take the messages received from the topic and pass them directly to the `_send_command` function, which handles communication with the haptic suit hardware. This replaces the locally generated command patterns in the existing controllers.
